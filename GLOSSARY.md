# GLOSSARY — sandbox-package-creator v2.6

本文件解释 SKILL.md 中出现的核心术语。按章节顺序排列，便于对照阅读。

---

## Shape 阶段

### 协作协议 (🤝)
SKILL.md 开篇的自检框架。不给人贴标签，而是让人通过四道开放式问题看清自己当前的状态——对需求有多大把握、更需要执行还是挑战、是否愿意被追问、最担心的是什么。AI 根据人的回答自动调校后续行为。

### 自检镜四问
协作协议的四个开放问题。每题都带「其他：____」选项，允许多元回答。人的答案决定 AI 在 Shape 阶段是「多翻译、执行优先」还是「多穷举、帮你挖出你本来就知道的东西」。

### 愿望 (1.1 节)
Shape 阶段的起点——先问「你真正想要什么体验」，再由 AI 把模糊愿望翻译成工程语言。不假设人一定懂技术，也不假设人一定说不清。

### 淘汰链 (1.2 节)
形态选择流程：「Skill → MCP → Sandbox Package → 不换形态深耕」。每走一步都问：这是什么动作类型？调用频率？数据流向？谁用？走到沙盒包后，**追问一句**：「不换形态，深耕能达到吗？」

### 反向追问 (淘汰链末尾)
淘汰链走到沙盒包后的最后一问。用途：防止人为了「换个形态」而逃避在现有框架里深耕的可能性。

### 兼容层级 (1.4 节)
四层接口标准阶梯：

| 层级 | 标识 | 适用场景 | 关键特征 |
|------|------|---------|---------|
| 🔴 原生 QuickJS | 纯文本进出 | 内部工具 | 最快，无结构化 |
| 🟡 Package API 统一包装 | `{success, data, error, elapsed_ms}` | 团队内协作 | 结构化返回 |
| 🟠 跨包调用 + 错误码 | `-1`~`-9` + `retry_trace` | 复杂系统 | 故障可定位 |
| 🟢 公开 API | 语义化版本 | 对外发布 | 向后兼容承诺 |

不需要一开始就选最顶层。从实际需要的层级开始，往上一层层加。

### 演进路线 (1.6 节)
从单线升级为**双分叉**：深耕（当前框架内优化）vs 重构（跨形态升级）。两条路都有各自的问题清单，让人看清每条路的代价再做选择。

## Explore → Diagnose → Confirm 阶段

### 三阶段回环
Shape 之后的验证链条。Explore（穷举可能方案）→ Diagnose（诊断每个方案的风险）→ Confirm（确认人对方案的理解）。任何一步发现偏差都可以退回上一步。

## Develop 阶段

### 统一返回包装 (5.1 节)
所有沙盒包函数的返回格式：
```js
{ success: true/false, data: ..., error: "...", elapsed_ms: ... }
```
目的：AI 调用方不需要猜测返回结构，直接判断 `success` 决定下一步。

### 错误码体系 (5.1 节)
`-1` 到 `-9` 的标准化错误码。每个数字对应一个故障类别（参数错误、超时、权限、网络等），AI 不需要解析错误文本就能判定问题类型。

### retry_trace (5.1 节)
故障追踪链。当跨包调用出错时，每一层都在返回中追加自己的错误信息，形成完整链路：「调用 A → A 调用 B → B 返回错误码 -3 → A 判定为下游超时」。AI 拿到 `retry_trace` 就能定位问题来源，决定是重试、降级还是报告人类。

### 按类型性能准则 (5.2 节)
四种沙盒包类型的独立性能策略：
- **搜索类**：Promise.all 并发 + 去重去噪 + 结果缓存
- **文件类**：流式读写，禁止整文件加载到内存
- **网络类**：超时必设 + 重试退避 + 响应压缩
- **数据处理**：懒加载 + 分页 + 增量计算

每种类型都有明确红线，不是「越多越好」，是「刚刚好」。

### QuickJS 性能陷阱 (5.3 节)
六条已知的 QuickJS 运行时坑：
1. JSON 循环引用 → 直接崩溃
2. 单次日志过多 → UI 线程阻塞
3. fetch 无超时 → 永久挂起
4. 同步循环中大量异步 → 内存堆积
5. 正则回溯爆炸 → CPU 卡死
6. 缓存设计缺失 → 每次都重新算

每条都有对应的黄金法则。

### 自测清单 (5.5 节)
Develop 完成后的 8 条验证项。比 v1.2 的 6 条增加了性能基准测试和接口一致性校验。

## 全局概念

### 同时决策
本 Skill 的隐含协作模式。不是「人类想完 AI 执行」，也不是「AI 推方案人类拍板」——是两个脑子同时想、互相触发。协作协议中的「其他：____」选项专为这种模糊边界设计。

### 多元回答
所有选择题都允许填写「其他：____」——不锁选项，不强制分类。框架帮人看清当下，人自己知道该让 AI 怎么配合。同一个人的不同任务可以走不同分支。

### 双分叉
演进路线的两个方向：深耕（在同一个形态里继续挖潜力）和重构（换一个形态重新开始）。没有哪个天生更好——Skill 提供两边的具体问题清单，让人自己做价值判断。
