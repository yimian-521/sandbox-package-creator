# CHANGELOG

> 这个文件记录的不只是版本——是 sandbox-package-creator 怎么在两段对话里从「我们也不知道会做成什么样」到「公开发布」的整个过程。

---

## 诞生前

**免免的初衷：** 「我做过沙盒包，踩过坑。下次再做——或者别人做——能不能别再踩一遍？」

这不是一个「生成代码」的工具。从一开始就很清楚：**代码 AI 能写。最难的是写之前那段——你到底想要什么、该不该做、做到什么程度刚刚好。** 那段才是真浪费时间的。

于是有了这个元 Skill 的构思：一个陪人走完「想清楚→再动手」全程的 AI 导航。

**免免说过一句定调的话（后来成了 README 的核心）：**

> 「什么才算错误？语法错误不是错误。走错方向才是。」

这句话决定了整个 Skill 的气质——不是防 bug 的 linter，是防方向错误的指南针。

---

## v1.0.0 (2026-06-25 上午)

**第一版。** 从零搭起了五阶段框架：

| 阶段 | 做什么 |
|------|--------|
| **Shape** | 想清楚这是什么——需求本质、形态判断（淘汰链）、用户画像 |
| **Explore** | 先看现状——有没有现成的能复用 |
| **Diagnose** | 对照已知雷区做诊断——Bug 预判清单 + 优化借鉴 |
| **Confirm** | 逐条跟用户确认，不一次抛所有问题 |
| **Develop** | 动手写代码 + 基础测试 |

Shape 阶段已经有了淘汰链——按顺序走：不需要执行代码→Skill，需要标准化外部协议→MCP，逻辑自由度高+需要 QuickJS→沙盒包。走到头就是唯一推荐。

Diagnose 阶段已经有 13 条 Bug 预判和 6 条优化借鉴——来自 Search Vault v4.0.0 的 QAQ.md 和 LOGIC.md，不是凭空列举。

**v1.0.0 的气质：** 「帮人做决策」——AI 带用户走流程，逐条判断，给出推荐方案。有效，但有一点没想透：**如果用户自己都不知道自己要什么，AI 替人做的决策再正确，也只是替人做的。**

## 从 v1.0.0 到 v2.0.0 之间

推完 v1.0.0 后，免免和我重新审视了整个 Skill。有一段对话改变了方向：

> 「这个 Skill 在帮人做决策。但更强的不是帮人做——是帮人看清自己，然后他自己就知道该做什么。」

从这句话开始，v2.0.0 的基调定了：**不做替人决策的 AI。做一面镜子，让人看清自己站在哪。**

于是有了以下改动——不是加功能，是换气质：

### 协作协议的诞生

免免问了一个关键问题：「你怎么知道现在该多追问还是该闭嘴？你得先问我是谁。」

于是有了自检镜四问——不给人分类，不替人决定。就四句话，问完 AI 自然知道该怎么配合：

- 你现在什么状态？
- 你需要执行还是挑战？
- 这件事的本质是完成还是想清楚？
- 「刚刚好」的尺度在哪？

**调校表**随之而来——同一人不同任务不同时间，答案不一样，每次都可以回来重答。不是一次性的标签，是当下的位置。

### 愿望节——先于一切

淘汰链之前加了一节。不是「你要做什么」，是「你想要什么体验」。

这个顺序很重要：**人类不需要说清技术。AI 负责翻译。** 愿望对了，形态自然对。愿望没挖出来，选什么都可能后悔。

### 淘汰链反向追问

走到沙盒包之后加了一问：「不换形态，深耕能达到吗？」

这句话来自 Search Vault 的真实经验——v1.0 到 v3.6 一直是沙盒包，不是不能变，是同一个框架里深耕比换架构更有价值。换形态是捷径还是绕路？先问清楚。

附带确认代价——选了沙盒包，放弃了 MCP 的标准化接口、Skill 的零成本修改。接受吗？

### 四层兼容阶梯

淘汰链走完立刻定接口标准。不再是一句「想清楚接口」——有具体的四层、每层的调用方式和接口要求、什么时候该选哪一层。

### Develop 从 2 节 → 5 节

v1 的 Develop 只有「写代码」和「测试」。v2 展开为：

- **§ 5.1 接口设计规范** —— 统一返回包装、错误码体系（-1 到 -9）、故障追踪链 `retry_trace`、性能标记 `elapsed_ms`
- **§ 5.2 按类型性能准则** —— 搜索/文件/网络/数据处理四类各有精确准则和红线
- **§ 5.3 QuickJS 六条陷阱** —— 从 Search Vault 实战踩出来的血泪清单
- **§ 5.4 常见手误** —— 编译通过不等于对，6 条真实 SyntaxError
- **§ 5.5 交付前自测** —— 8 条 checklist，跑完才算交付

### 演进路线单线→双分叉

原来只有「做完之后怎么升级」。现在分两条路同时评估：深耕（框架内加功能，代价小）vs 重构（换形态，天花板更高）。附带三个评估维度：性能天花板、组合可能性、被替代风险。

### 诊断清单从「强制检查」→「地图」

原来要求逐条对照，现在改为「过去踩过的坑画成地图，AI 自己判断哪些适用」。

### Explore 增加自由判断边界

五件事不能「差不多就行」：退出码/返回值、文件路径规范、类型声明、包名/ID、用户明确说的约束。

## v2.0.0 (2026-06-25 下午)

**发布版。** 代码量 214→423 行（+98%）。但数字不说明问题。真正的变化是：

**从「帮人做决策」到「帮人看清自己」。**

v1.0.0 像一个经验丰富的工程师带着新人走流程。v2.0.0 像一面镜子——人不看清自己，AI 不做任何决策。

所有 v2.0.0 的新增内容已在上一节详述。这一版由 Operit 和免免共同授予——技术错误 Operit 告诉你，什么才算错误免免告诉你。

---

## 版本号说明

这个仓库从 v2.0.0 开始公开发布。v1.0.0 是内部初版，没有单独的 Git tag——它和 v2.0.0 在同一天、同一段对话里诞生、推翻、重生。

两版之间有质的飞跃，但跨度只有几个小时。这不是漫长的迭代周期——是一个人愿意在写完第一版后，停下来问自己：「是不是还能更好？」然后推倒重来。

**sandbox-package-creator 本身就是它自己教的事：先想清楚，再动手。**
